## 1 项目定位

**一句话**：手机端作为遥控器，向车端 AI 大脑发送用户意图 JSON，车端推理后编排桌面卡片。

面向管理层演示三个核心体验：

| # | 场景 | 用户行为 | 车端预期 |
|---|------|---------|---------|
| 1 | 娱乐需求响应 | 输入"不想听音乐了，我想听播客" | 喜马拉雅推荐卡移入堆栈顶 |
| 2 | 上车智能推荐 | 选择用户 + 行程 + 触发上车 | 根据 persona × trip × time 推荐最佳卡片 |
| 3 | AI 播客生成 | 发送内容链接 | 调用豆包生成播客 → 播客卡移入堆栈顶 |

## 2 系统架构

### 2.1 总体架构

```
┌─────────────┐          ┌──────────────────────┐          ┌──────────┐
│  手机端 H5   │←──WSS──→│   阿里云服务器         │←──WSS──→│  车端 AI  │
│  Netlify     │          │                      │          │  Android  │
└─────────────┘          │  relay.js (:9000)     │          └──────────┘
       │                 │  podcast-api.js(:9001)│
       │── POST ────────→│  Nginx (:8443 SSL)    │
       │  /api/podcast   │  audio/ (mp3 存储)    │
       │                 │  podcast-store.json   │
       │                 └──────────────────────┘
       │                          │
       │                          │── WSS + Header ──→ 豆包 API
       │                          │   (播客生成代理)
```

**线上地址**：

| 服务 | URL |
|------|-----|

| 中继 WSS | wss://zhangchang.duckdns.org:8443/ws?room=car_001 |
| 播客 API | https://zhangchang.duckdns.org:8443/api/podcast |
| 播客列表 | https://zhangchang.duckdns.org:8443/api/podcast/list |

### 2.2 各端职责

| 端 | 负责 | 不负责 |
|----|------|-------|
| **手机端** | 用户身份切换、文字输入、AI 播客触发、上车触发、JSON 组装发送 | 模型推理、卡片渲染、音频播放 |
| **中继服务器** | WebSocket 房间广播、消息转发、豆包 API 代理（播客生成） | 业务逻辑决策、消息持久化 |
| **车端** | 接收 JSON → LLM 推理 → 卡片编排 → WebView 渲染 → 音频播放 → ACK | 用户输入收集、播客生成 |

### 2.3 通信协议

| 链路 | 协议 | 说明 |
|------|------|------|
| 手机 ↔ 中继 | WebSocket (WSS) | JSON 消息双向通信 |
| 车端 ↔ 中继 | WebSocket (WSS) | 同一房间，消息互转 |
| 中继 → 豆包 API | WebSocket Binary | 自定义二进制协议 v3（播客生成代理） |

## 3 AI 播客播放链路（核心链路）

### 3.1 性能基线（实测数据 2026-03-28）

7 篇微信公众号长文的完整生成测试：

| 文件 | 分类 | 大小 | 轮数 | 首token | 首段音频 | 生成耗时 | 播客时长 | Token |
|------|------|------|------|---------|---------|---------|---------|-------|
| ai_tech_01 | AI科技 | 12.7MB | 69 | 1s | 14s | 415s | 18.5min | 32,191 |
| ai_tech_02 | AI科技 | 5.7MB | 37 | 5s | 19s | 187s | 8.3min | 15,347 |
| people_01 | 人物 | 3.4MB | 25 | 8s | 19s | 140s | 5.0min | 9,395 |
| people_02 | 人物 | 10.0MB | 71 | 10s | 19s | 332s | 14.5min | 25,730 |
| biz_01 | 商业管理 | 7.1MB | 51 | 9s | 24s | 235s | 10.3min | 17,868 |
| biz_02 | 商业管理 | 5.1MB | 35 | 8s | 19s | 163s | 7.5min | 13,621 |
| biz_03 | 商业管理 | 19.8MB | 103 | 1s | 14s | 591s | 28.8min | 51,552 |
| **合计** | | **63.8MB** | **391** | **avg 6s** | **avg 18s** | **34.4min** | **92.9min** | **165,704** |

**关键指标**：

| 指标 | 值 | 含义 |
|------|-----|------|
| 首 token 延迟 | 1-10s（avg 6s） | 豆包 API 解析文章 + 首段文本生成 |
| 首段可播音频 | 14-24s（avg 18s） | 用户发出链接后最早能听到声音的时间 |
| 生成速度比 | **2.7× 实时** | 1 分钟播客只需 22s 生成，远快于播放速度 |
| 音频格式 | MPEG2 L3, 96kbps, 24kHz mono | 豆包 API 固定输出 |
| audio_url 返回时机 | PodcastEnd 事件（全部生成完成后） | 不可提前获取 |
| audio_url 有效期 | 24 小时 | 豆包 CDN 签名 URL |
| 流式块大小 | ~4.6KB/chunk | RoundResp(361) 事件 |

### 3.2 播放链路设计（MVP）：等待完成 + CDN URL

**MVP 策略**：手机端等待豆包 API 生成完毕（PodcastEnd 事件），拿到 `cdn_url` 后推送给车端播放。简单可靠，无需服务端额外处理音频流。

#### 时序图

```
用户        手机端 H5       服务端          豆包 API          车端
 │           │              │               │                │
 │─ 发送URL ─→│              │               │                │
 │           │── WebSocket ──────────────────→│                │
 │           │              │               │                │
 │  "链接已收到！│             │               │                │
 │  正在生成..." │             │               │                │
 │           │              │               │                │
 │           │←── 首 token ─────────────────│                │
 │  "已开始生成，│             │               │                │
 │  预计X分钟"  │             │               │                │
 │           │              │               │                │
 │           │←── 流式音频块 ×N ────────────│                │
 │           │   (仅用于进度计算,不中转)      │                │
 │           │              │               │                │
 │           │←── PodcastEnd(363) ─────────│                │
 │           │      含 cdn_url              │                │
 │           │              │               │                │
 │  "生成完成！" │              │               │                │
 │           │── POST postcard JSON ───────→│               │
 │           │              │── push ── postcard JSON ─────→│
 │           │              │               │         ┌──────┤
 │           │              │               │         │ 播放 │
 │           │              │               │         │cdn_url│
 │           │              │               │         └──────┤
 │           │←── ACK ─────────────────────────────────────│
```

#### 三阶段详解

**阶段 1：建连与启动（0-6s）**
1. 用户在手机端选择/输入 URL
2. 手机端直连豆包 WebSocket API（StartConnection → StartSession）
3. 手机端 UI 显示 **"链接已收到！正在生成播客..."**（loader 图标，黄色背景 `#FEF3C7`）

**阶段 2：生成中 — 持续反馈（6s - 完成）**
1. 收到首 token（avg 6s）→ UI 刷新为 **"已开始生成，预计X分钟，完成后会提醒您"**
2. 时长预估公式：`预估分钟 = round(原文字数 / 800)`（基于实测 ~800 字/分钟播客时长）
3. 手机端持续接收流式音频块（RoundResp 361），仅用于监控进度，**不做中转**
4. 可选：根据已接收音频大小更新"已生成 X%"进度

**阶段 3：完成与推送（PodcastEnd 时刻）**
1. 收到 PodcastEnd(363) 事件 → 提取 `meta_info.audio_url`（即 cdn_url）
2. 手机端 UI 刷新为 **"生成完成！请在车端查看播客详情"**（check-circle 图标，绿色背景 `#ECFDF5`）
3. 组装 postcard JSON（含 `cdn_url`）→ POST 至服务端 → 透传车端
4. 车端收到后用 `cdn_url` 开始播放

#### 用户等待时间

| 内容长度 | 预估播客时长 | 生成等待时间 | 说明 |
|---------|------------|------------|------|
| 短文（~2000字） | 5 min | ~2.5 min | 快速体验 |
| 中文（~5000字） | 10 min | ~4 min | 典型微信长文 |
| 长文（~10000字） | 20 min | ~7 min | 深度文章 |
| 超长文（~25000字） | 30 min | ~10 min | 最长边界 |

> ⚠️ MVP 阶段用户需等待全部生成完成才能播放。**演示时优先使用预设播客**（0s 等待）。

### 3.3 P1 未来优化：流式中转方案

> 以下方案记录为 P1 优先级，待车端播放能力确认后实施。

**核心洞察**：生成速度 2.7× > 播放速度 1×，意味着边生成边播放时，缓冲区只会越来越满，不会断流。

**方案概要**：
1. 手机端接收豆包流式音频块 → 实时转发给服务端
2. 服务端拼接临时 mp3 → 累积 0.5MB（~45s 播放时长）后生成播放 URL
3. 服务端推送 postcard JSON（含 `stream_url`）→ 车端开始边下边播
4. **用户等待时间从"全部生成完"缩短至 ~18s**

**前提条件**：
- 车端 MediaPlayer 支持渐进式下载（HTTP Range 请求）
- 服务端需新增音频流接收/拼接/托管能力

**P1 实施时**需新增的服务端 API：`/api/podcast/start`、`/api/podcast/chunk`、`/api/podcast/done`

### 3.4 预设播客 vs 实时生成

| | 预设播客 | 实时生成 |
|--|--------|---------|
| **触发** | 用户从预设列表选择 | 用户输入自定义 URL |
| **音频来源** | 提前批量生成，存 CDN/服务端 | 实时调豆包 API |
| **用户等待** | 0s（直接播放） | 2-10 分钟（等待全部生成） |
| **JSON payload** | `source.type: "preset"`, 直接带 `podcast.cdn_url` | `source.type: "url"`, 等 PodcastEnd 拿 `cdn_url` |
| **MVP 建议** | **演示时优先用预设内容** | 作为"自由输入"能力展示 |

## 4 手机端功能详述

### 4.1 页面结构

单页面对话式 UI，390×844 逻辑像素（iPhone 尺寸）。

```
┌──────────────────────────────────────┐
│  娱乐Agent手机入口    [A] 用户A ∨    │  ← 顶栏：标题 + 用户切换胶囊
├──────────────────────────────────────┤
│                                      │
│         [ 对话消息区域 ]              │  ← 消息气泡 + 系统反馈卡片
│                                      │
├──────────────────────────────────────┤
│  [ 输入需求...                  ↑ ]  │  ← 输入框 + 发送按钮
│  [ 模拟上车 ]    [ AI播客 ]          │  ← 两个功能触发按钮
└──────────────────────────────────────┘
```

### 4.2 六条交互链路

#### 链路 A：切换用户

**触发**：点击顶栏右上角用户胶囊（头像 + 名字 + ∨）
**弹窗**：「选择用户身份」底部弹窗
**选项**：

| persona_id | 标签 | 描述 |
|------------|------|------|
| `user_a` | 用户A：欧美流行重度发烧友 | 欧美流行、电子、Hip-hop |
| `user_b` | 用户B：喜欢听国语民谣、轻音乐 | 华语民谣、轻音乐、Acoustic |
| `user_c` | 用户C：喜欢听古典乐、播客 | 古典乐、知识播客、有声书 |

**行为**：仅更新本地状态，不发送 JSON。选中后关闭弹窗，顶栏胶囊更新。

**对应设计稿**：Screen 2（选择用户）、Screen 3（切换后状态）

#### 链路 B：文字输入 → content JSON

**触发**：在输入框键入文字 → 点击发送按钮（↑）
**流程**：
1. 消息区显示用户消息气泡（紫色，靠右）
2. 组装 `type: "content"` JSON → HTTP POST 至服务端
3. 收到 ACK 后，消息区显示系统反馈卡片：
   - 蓝色 `content` 标签 + "已发送" 状态
   - 反馈文案（如"你的偏好已经更新！请在车端确认变化"）
   - 左下角 `● 已发送至车端` 状态指示
   - 右下角「查看JSON」按钮

**对应设计稿**：Screen 4（content 回复）

#### 链路 C：触发上车 → recommend JSON

**触发**：点击底部「模拟上车」按钮
**弹窗**：「模拟上车信息」底部弹窗
**表单字段**：

| 字段 | 控件 | 值 |
|------|------|-----|
| 行程类型 | Chip 单选 | 早上通勤 / 下班回家 / 周末出游 / 情侣时光 / 儿童 |
| 行程时间 | 数字输入框 | 默认 30，单位"分钟" |

**行程类型枚举**：

| scene_id | scene_label |
|----------|-------------|
| `morning_commute` | 早上通勤 |
| `commute_home` | 下班回家 |
| `weekend_outing` | 周末出游 |
| `couple_time` | 情侣时光 |
| `kid_mode` | 儿童 |

**流程**：
1. 选择行程类型 + 输入时长 → 点击「触发上车信号」
2. 消息区显示系统消息"→ 触发上车事件 →"
3. 组装 `type: "recommend"` JSON（自动填充 `time_context`）→ POST
4. 收到 ACK → 显示绿色 `recommend` 标签反馈卡

**对应设计稿**：Screen 9（上车弹窗）、Screen 10（recommend 回复）

#### 链路 D：AI 播客 - 选择内容

**触发**：点击底部「AI播客」按钮
**弹窗**：「选择AI播客内容」底部弹窗
**内容**：
- 4 张预设卡片（按当前 persona 动态展示），每张含标题 + 描述
- 底部输入框："输入自己想听的（请输入URL）" + 「生成」按钮

**对应设计稿**：Screen 6（AI 播客选择）

#### 链路 E：AI 播客 - 生成与播放

**触发**：在链路 D 中选择预设卡片或输入自定义 URL

**预设播客路径**（0s 等待）：
1. 消息区显示 "生成完成！请在车端查看播客详情"
2. 直接组装 postcard JSON（预设的 `cdn_url`）→ POST → 车端播放

**实时生成路径**（渐进式状态反馈）：
1. 消息区显示用户消息气泡（URL）
2. 状态提示 ①：**"链接已收到！正在生成播客..."**（loader 图标，黄色背景 `#FEF3C7`）
3. 手机端直连豆包 WebSocket（StartConnection → StartSession）
4. 收到首 token → 状态提示 ②：**"已开始生成，预计X分钟，完成后会提醒您"**（loader 图标，黄色背景）
5. 持续接收流式音频块（仅监控进度，不中转）
6. PodcastEnd → 提取 `cdn_url` → 状态提示 ③：**"生成完成！请在车端查看播客详情"**（check-circle 图标，绿色背景 `#ECFDF5`）
7. 组装 postcard JSON（含 `cdn_url`）→ POST 至服务端 → 透传车端 → 车端播放
8. 收到 ACK → 显示紫色 `postcard` 标签反馈卡

**对应设计稿**：Screen 7（生成中）、Screen 8（postcard 回复）

#### 链路 F：查看 JSON（调试工具）

**触发**：在任何反馈卡片中点击「查看JSON」
**弹窗**：「用户传来JSON」底部弹窗，深色背景 + 格式化 JSON

**对应设计稿**：Screen 5（查看 JSON）

## 6 豆包播客 API 集成

### 6.1 接口概要

| 项 | 值 |
|----|-----|
| 协议 | WebSocket 二进制协议 v3 |
| 地址 | `wss://openspeech.bytedance.com/api/v3/sami/podcasttts` |
| 鉴权 | Header: `X-Api-App-Id`, `X-Api-Access-Key`, `X-Api-Resource-Id`, `X-Api-App-Key` |
| Resource ID | `volc.service_type.10050` |

### 6.2 二进制协议

```
帧结构（Pre-connection）: header(4) + event_type(4) + payload_size(4) + payload_json
帧结构（Post-connection）: header(4) + event_type(4) + sid_len(4) + session_id + payload_size(4) + payload_json
Header 固定: [0x11, 0x14, 0x10, 0x00]
```

### 6.3 握手流程

```
手机端                              豆包 API
  │── StartConnection(event=1) ────→│
  │←── ConnectionStarted(event=50) ─│  → 获得 session_id
  │── StartSession(event=100) ─────→│  → 携带所有播客参数
  │←── SessionStarted(event=150) ───│
  │  ┌── 流式循环 ──────────────────┐
  │←─┤ 360: RoundStart (text)      │
  │←─┤ 361: RoundResp (audio)      │  → ~4.6KB/chunk，手机端用于进度监控
  │←─┤ 362: RoundEnd              │
  │  └─────────────────────────────┘
  │←── 363: PodcastEnd ────────────│  → meta_info.audio_url（CDN 完整 URL）
  │←── 152: SessionFinished ───────│
  │── FinishConnection(event=2) ──→│
```

### 6.4 两种输入模式

**模式 A — input_text（实时演示用，短文本 < 200 字）**
```jsonc
{
  "input_text": "文本内容...",  // ⚠️ 顶层
  "audio_config": { "format": "mp3", "sample_rate": 24000, "speech_rate": 0 },
  "speaker_info": { "random_order": true, "speakers": ["zh_male_dayixiansheng_v2_saturn_bigtts", "zh_female_mizaitongxue_v2_saturn_bigtts"] }
}
```

**模式 B — input_url（URL 文章模式）**
```jsonc
{
  "input_info": { "input_url": "https://...", "return_audio_url": true },  // ⚠️ 在 input_info 内
  "use_head_music": true, "use_tail_music": false,
  "audio_config": { "format": "mp3", "sample_rate": 24000, "speech_rate": 0 },
  "speaker_info": { ... }
}
```

### 6.5 错误处理

| 场景 | 手机端行为 |
|------|-----------|
| 建连失败 | Toast "播客服务暂时不可用" |
| 生成中断连 | 自动重试一次（retry_info），失败则提示 |
| 内容过滤 (50302102) | 提示 "该内容暂不支持生成播客" |
| audio_url 过期（24h） | 提示重新生成 |

## 12 播客内容存储

### 12.1 存储结构

```
/opt/relay/
├── podcast-store.json     # 播客索引（预设 + 用户生成）
├── audio/
│   ├── ai_tech_01.mp3     # 预设音频
│   ├── gen_17747xxx.mp3   # 用户生成的音频
│   └── ...
```

### 12.2 数据模型

```json
{
  "id": "gen_1774767112003",
  "source_url": "https://mp.weixin.qq.com/s/xxx",
  "title": "小米在人工智能和机器人领域",
  "desc": "小米最近的AI技术进展...",
  "audio_file": "gen_1774767112003.mp3",
  "duration_sec": 682,
  "created_at": "2026-03-29T14:00:00Z",
  "source": "generated"
}
```

### 12.3 去重机制

同一 `source_url` 不会重复生成。手机端发送已有 URL → 服务端直接返回缓存（0 token，0 等待）。

### 12.4 音频持久化

豆包 CDN URL 24 小时过期。服务端在生成完成后立即下载 mp3 到本地，车端拿到的是服务端永久 URL。
